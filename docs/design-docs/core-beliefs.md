# Song Maker — Core Beliefs

## 1. 설계 신념

이 프로젝트의 모든 기술적 의사결정은 아래 신념에 기반한다.

---

### 자동화는 사용자의 판단을 대체하지 않는다

곡의 방향(장르, 분위기, 주제)은 반드시 사용자가 결정한다. 시스템은 데이터를 제공하고 실행을 자동화할 뿐, 창작 의사결정을 대신하지 않는다.

**적용 예시:**
- 트렌드를 자동 분석하되, 장르를 자동 선택하지 않는다
- 곡을 자동 생성하되, 업로드 공개 여부는 사용자가 결정한다
- 이미지를 자동 생성하되, 재생성 옵션을 항상 제공한다

---

### 파이프라인은 끊어질 수 있다

어떤 Stage에서든 실패할 수 있고, 실패는 정상이다. 중요한 것은 실패 지점에서 재개할 수 있는 것이다.

**적용 예시:**
- 각 Stage 완료 시 상태를 파일에 기록한다
- 개별 Stage를 독립 실행할 수 있다
- 검증 게이트에서 실패 시 사용자에게 선택지를 준다 (재시도/건너뛰기/중단)

---

### 외부 API는 언제든 바뀔 수 있다

Suno API는 비공식이다. Gemini 모델은 프리뷰다. YouTube API는 할당량이 변한다. 어떤 외부 API든 변경/중단될 수 있다.

**적용 예시:**
- 외부 API 호출은 어댑터 모듈로 감싼다
- API 변경 시 어댑터만 교체하면 나머지 코드는 영향받지 않는다
- 하드코딩된 모델명/엔드포인트 없이 config.toml에서 관리한다

---

### 단순함이 기능보다 우선한다

CLI 도구이다. 웹 UI, 사용자 계정, 실시간 스트리밍은 만들지 않는다. 하나의 파이프라인을 잘 실행하는 것이 목표이다.

**적용 예시:**
- Typer + Rich로 충분하다. 웹 프레임워크를 도입하지 않는다
- JSON 파일로 충분하다. 데이터베이스를 도입하지 않는다
- `songmaker run` 하나로 전체 파이프라인이 실행된다

---

### 검증 없이 다음 단계로 넘어가지 않는다

각 Stage 사이에 검증 게이트가 있다. 출력물이 기준을 통과해야만 다음 Stage로 진행한다. 이것이 하네스 구조의 핵심이다.

**적용 예시:**
- 오디오 파일 크기와 무결성을 검증한 후 이미지 생성으로 넘어간다
- 영상 스트림을 검증한 후 업로드로 넘어간다
- 검증 결과를 meta.json에 기록하여 추적 가능하다

---

## 2. 하지 않는 것 (Anti-Patterns)

| 하지 않는 것 | 이유 |
|-------------|------|
| 곡 방향 자동 결정 | 사용자의 창작 의도를 존중 |
| 검증 없이 Stage 연결 | 하네스 구조 위반, 오류 전파 |
| API 직접 호출 (어댑터 없이) | API 변경 시 전체 코드 수정 필요 |
| 웹 UI 추가 | 범위 초과, 복잡성 증가 |
| 자동 공개 업로드 | 저작권/품질 미확인 상태에서 공개 위험 |
| 에러 무시 후 진행 | 손상된 파일이 파이프라인 하류로 전파 |
